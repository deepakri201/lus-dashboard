# LUS Inter-Rater Reliability Dashboard

A browser-based dashboard for reviewing lung ultrasound (LUS) annotation agreement between two raters. It connects to a Google Cloud Storage (GCS) bucket, loads annotation statistics, and opens side-by-side PNG comparisons of the max-scoring frames per metric and lung zone.

<img width="1512" height="815" alt="lus_dashboard" src="https://github.com/user-attachments/assets/ee896934-df31-4572-8b50-8742de127ec7" />

## What the dashboard does

- **Phase / patient / lung zone filters** — narrow the cohort before analysis.
- **Two rater selectors** — compare any pair of annotators in the bucket.
- **IRR charts** — scatter plots for pleura %, discrete B-lines, and confluent B-lines (Pearson *r* / Cohen’s κ). Click a point to open PNGs for that case or zone.
- **Case metrics table** — max pleura %, discrete B-line count, and confluent B-line count per rater, with variance highlights.
- **PNG comparison viewer** (`png_viewer.html`) — per metric, side-by-side images for rater A vs rater B, one row per lung zone, with frame number and score overlaid on each image.

### Data sources (after Google sign-in)

| Mode | Reads from GCS |
|------|----------------|
| **Pre-computed stats (CSV)** | `stats/annotations_by_clip.csv`, `stats/irr_data.csv` |
| **Calculate from annotations (JSON)** | `completed-assignments-all/` (manual LUS score JSON files) |

PNG images are always loaded from `pngs/` in the same bucket (see pipeline below).

## Repository contents

This repo is meant to hold the **dashboard UI and extraction tooling**. Clinical data stays in GCS, not in GitHub.

| Path | Purpose |
|------|---------|
| `lus_annotation_dashboard.html` | Main dashboard (single HTML file) |
| `png_viewer.html` | PNG comparison viewer (opened from the dashboard) |
| `scripts/extract_max_frame_pngs.py` | Build max-frame PNGs from DICOM + JSON |
| `requirements.txt` | Python dependencies for the extraction script |

Do **not** commit patient data, DICOM files, or large CSV exports unless your repo policy explicitly allows it.

## GCS bucket layout

Expected bucket structure (example bucket name: `deepa_dashboard`):

```
gs://deepa_dashboard/
├── completed-assignments-all/   # DICOM + manual_lus_score.json (source annotations)
├── stats/                       # Pre-computed CSVs (optional but recommended)
│   ├── annotations_by_clip.csv
│   └── irr_data.csv
└── pngs/                        # Max-frame PNGs (generated by extraction script)
    └── phase-N/annotator-folder/case-id/{file_hash}_{pleura|discrete|confluent}.png
```

## Step 1 — Generate PNGs from DICOM + JSON

The dashboard PNG viewer expects pre-rendered frames under `pngs/`. Generate them locally from the same assignment tree used in `completed-assignments-all/`.

### Install Python dependencies

```bash
pip install -r requirements.txt
```

`requirements.txt` includes:

- `pydicom`, `Pillow`, `numpy` — DICOM read and image export  
- `pylibjpeg`, `pylibjpeg-libjpeg` — JPEG-compressed ultrasound DICOM  
- `matplotlib` — optional preview during testing  

### Run extraction

Place (or symlink) `completed-assignments-all/` next to the script’s working directory, then:

```bash
# Quick test on one default clip
python scripts/extract_max_frame_pngs.py --test

# Full run (all phases)
python scripts/extract_max_frame_pngs.py --jobs 4

# Single phase
python scripts/extract_max_frame_pngs.py --phase phase-5 --jobs 4 --verbose
```

For each clip with `frame_annotations`, the script writes up to three PNGs (only for metrics that were annotated):

- `{file_hash}_pleura.png` — frame with max pleura %
- `{file_hash}_discrete.png` — frame with max discrete B-line count
- `{file_hash}_confluent.png` — frame with max confluent B-line count

Output mirrors the assignments path under `pngs/`.

### Upload PNGs to GCS

```bash
gsutil -m cp -r pngs gs://deepa_dashboard/
```

Re-run extraction and upload when annotations change.

## Step 2 — Use the dashboard (HTML files)

The dashboard is **two static HTML files**. They must be served over **HTTP** (not opened as `file://`). Google sign-in and GCS API calls require a real origin such as `http://localhost:8000`.

### Get the HTML files

**Option A — Clone this repo**

```bash
git clone <your-repo-url>
cd <repo-directory>
```

**Option B — Download from GitHub**

Download `lus_annotation_dashboard.html` and `png_viewer.html` into the **same folder**. Both files are required; the dashboard opens `png_viewer.html` in a new tab for PNG comparisons.

### Configure the dashboard (`lus_annotation_dashboard.html`)

Before first use, open `lus_annotation_dashboard.html` in a text editor and set the constants at the top of the `<script>` block (search for `GCP SYSTEM CONFIGURATION CONSTANTS`). These must match **your** GCP project and bucket layout:

| Constant | What to set |
|----------|-------------|
| **`GCP_CLIENT_ID`** | OAuth 2.0 Client ID from [Google Cloud Console → Credentials](https://console.cloud.google.com/apis/credentials) (Web application type). Required for **Sign in with Google**. |
| **`GCS_BUCKET_NAME`** | Your GCS bucket name, e.g. `deepa_dashboard` (no `gs://` prefix). |
| **`GCS_ASSIGNMENTS_ROOT`** | Top-level folder for DICOM + JSON annotations, e.g. `completed-assignments-all`. Used when data source is **Calculate from annotations (JSON)**. |
| **`GCS_STATS_ROOT`** | Folder for pre-computed CSVs, e.g. `stats`. Used when data source is **Pre-computed stats (CSV)**. |
| **`GCS_PNGS_ROOT`** | Folder for max-frame PNGs, e.g. `pngs`. Must match the output of `extract_max_frame_pngs.py`. |
| **`ALLOWED_EMAIL_DOMAIN`** | Optional. e.g. `@hospital.org` to restrict sign-in to that email suffix. Leave `""` to allow any Google account (bucket IAM still controls data access). |

Example (edit to your values):

```javascript
const GCP_CLIENT_ID = "YOUR_CLIENT_ID.apps.googleusercontent.com";
const GCS_BUCKET_NAME = "your-bucket-name";
const GCS_ASSIGNMENTS_ROOT = "completed-assignments-all";
const GCS_STATS_ROOT = "stats";
const GCS_PNGS_ROOT = "pngs";
const ALLOWED_EMAIL_DOMAIN = "";   // or "@your.org"
```

If any of these do not match your bucket, sign-in may work but loading stats or PNGs will fail with 404/403 errors.

`png_viewer.html` does not need separate GCS constants — the dashboard passes bucket name, paths, and auth token in the URL when opening the viewer.

### Configure Google OAuth (one-time, project admin)

1. Create or select an **OAuth 2.0 Client ID** (Web application) in Cloud Console and copy the Client ID into **`GCP_CLIENT_ID`** above.
2. Under **Authorized JavaScript origins**, add every URL where you serve the HTML, for example:
   - `http://localhost:8000` (local dev)
   - `https://YOUR-ORG.github.io` (if using GitHub Pages)

Teammates need **read access** to the bucket (`storage.objects.get` and list on your bucket). Sign-in alone is not enough; GCS IAM controls who sees data.

### Launch locally

From the directory containing both HTML files:

```bash
python3 -m http.server 8000
```

Open in a browser:

```
http://localhost:8000/lus_annotation_dashboard.html
```

1. Choose a **data source** (CSV or JSON).
2. Click **Sign in with Google** (not Demo Mode for real data).
3. Select phase, optional patient/zone, and raters.

**Demo Mode** uses local mock data or local `stats/` CSV files only — use it for UI testing without GCS.

### Optional — GitHub Pages

You can host the two HTML files on GitHub Pages for a shared team URL. You still need to:

- Add the GitHub Pages origin to **Authorized JavaScript origins** in the OAuth client.
- Configure **bucket CORS** to allow that origin for Storage API requests.
- Grant bucket IAM to each teammate.

---

## Security notes

- **`GCP_CLIENT_ID`** is a public OAuth app identifier, not a secret. Bucket data is protected by Google sign-in and bucket IAM.
- Do not commit service account keys, `.env` secrets, or patient identifiers.
- The OAuth client ID in the HTML only works from origins you list in Cloud Console.
